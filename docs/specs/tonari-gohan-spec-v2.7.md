# となりごはん 仕様書 v2.7 — Webアプリ風MVP・広告検証版

最終更新: 2026-06-11
位置づけ: v2.5（LP → LINE手動検証）の Phase 0 定義を置き換える。
Phase 0 は「広告から直接触れるWebアプリ風MVP」として作る。

v2.5 からの主な変更:

- LP単体・LINE手動検証は Phase 0 の主導線から外す
- 検証対象を「LINE登録意欲」から「料理カードを見たときの『これなら作れそう』という反応」に変更
- 成功指標の中心は `click_want_to_make`（作ってみたいボタン）
- LINEは完全否定ではない。通知・継続利用の検証が必要になった段階で、補助導線として再検討する

---

## 1. コンセプト

### 1.1 一文定義

**となりごはんは、ユーザーがすでに作れる料理を起点に、少し変えれば作れそうな次の料理を提案するレパートリー拡張アプリである。**

### 1.2 コアコピー

- 第一候補: 「いつもの料理から、次の一品へ。」
- 第二候補: 「作れる料理の、すぐとなり。」
- 第三候補: 「レシピを探す前に、自分が作れそうな料理を見つける。」

### 1.3 解決する課題

自炊している人は、まったく料理ができないわけではない。ただし、

- いつも同じ料理になる
- レシピ検索しても遠すぎる
- 新しい料理を作る心理的ハードルが高い
- 食材・味付け・工程が少し違うだけなら作れそうなのに、その候補を見つけにくい

となりごはんは、一般的なレシピ検索ではなく、**今の自分の料理スキルから近い料理を出す**ことに特化する。

### 1.4 プロダクトの核

ユーザー体験はこの3ステップに絞る。

1. 作れる料理を選ぶ
2. 近い料理カードが出る
3. 「作ってみたい」として保存する

Phase 0では、この体験だけを成立させる。

---

## 2. ターゲット

### 2.1 メインターゲット

**自炊はしているが、レパートリーが固定化している人。**

- 一人暮らし
- 大学生、大学院生、若手社会人
- 週に数回は自炊する
- カレー、麻婆豆腐、唐揚げ、パスタ、炒め物などは作れる
- ただし新しい料理を探すのが面倒
- レシピアプリを見ても、結局いつもの料理に戻る

### 2.2 Phase 0で狙うユーザー

> 自炊初心者ではなく、定番料理が3〜5個ある人。
> ただし、その先の広げ方がわからない人。

### 2.3 狙わないユーザー

完全な料理初心者 / 本格料理を作りたい人 / 栄養管理をしたい人 / 冷蔵庫管理をしたい人 / 献立を一週間分作りたい人 / プロのレシピを細かく比較したい人 / ダイエット・筋トレ食を探したい人。

---

## 3. Phase 0で作るもの

### 3.1 Phase 0の目的

> ユーザーは、自分が作れる料理を選んだあとに出てくる「近い料理カード」を見て、作ってみたいと思うか？

見るべきなのは、LPの登録率ではなく、**プロダクト体験への反応**である。

### 3.2 作る範囲

- トップ画面
- 作れる料理選択
- おすすめ料理カード一覧
- 料理カード詳細
- 「作ってみたい」ボタン
- 保存一覧
- ローカル保存
- ダミーデータ
- スマホ最適化UI
- 最低限のアクセス解析・イベント計測

### 3.3 技術的前提

- Next.js / React（現行Web実装を前提）
- 思想としては「Webページではなく、アプリの試作品」
- DBなし、ログインなし。ユーザー状態は `localStorage` に保存

---

## 4. Phase 0で作らないもの

ログイン / ユーザーアカウント / DB / Supabase / Firebase / LINE API / LIFF / LINE通知 / 自動リマインド / 課金 / AI生成レシピ / ユーザー投稿 / 料理写真アップロード / 本格的なレシピ検索 / ネイティブiOS・Androidアプリ / 複雑なレパートリーマップ / 冷蔵庫管理 / 栄養計算 / 一週間献立。

判断基準: 便利機能・継続機能・パーソナライズ機能・データ管理機能は後回し。まず見るべきは「料理の近さ提案そのものが刺さるか」だけ。

## 5. 画面構成

### 5.1 全体構成

1. トップ画面
2. 作れる料理選択画面
3. おすすめカード一覧画面
4. 料理カード詳細画面
5. 作ってみたいリスト画面

実装上は1ページでもよい。ただし、ユーザー体験としては画面遷移しているように見せる。

### 5.2 トップ画面

目的: 広告から来たユーザーに、即座に「自分の作れる料理から広げるアプリなんだ」と理解させる。

表示内容: キャッチコピー / 短い説明 / 「作れる料理を選ぶ」ボタン / サンプルカード数枚 / スマホアプリ風の見た目。

```text
いつもの料理から、次の一品へ。

作れる料理を選ぶだけ。
少し変えれば作れそうな料理を、となりごはんが提案します。

[作れる料理を選ぶ]
```

### 5.3 作れる料理選択画面

目的: ユーザーに自分の料理経験を入力させる。テキスト入力ではなく選択式にする。

```text
作れる料理を選んでください

よく作る料理を3つくらい選ぶと、
あなたに近い料理が出てきます。
```

初期候補（チップ表示）: 麻婆豆腐 / 唐揚げ / カレー / 親子丼 / 豚キムチ / 鶏キムチ / 焼きそば / チャーハン / パスタ / グラタン / オムライス / 生姜焼き / 味噌汁 / 肉じゃが / ハンバーグ / 野菜炒め

選択ルール:

- 複数選択可。推奨は3〜5個
- 1個だけでも次に進める。選択数0なら進めない
- 選んだ料理は画面上部に固定表示

### 5.4 おすすめカード一覧画面

目的: 選んだ料理を起点に、近い料理を出す。**ここがPhase 0の中核。**

単に料理名を出すのではなく、「なぜ作れそうなのか」を必ず見せる。

カード例:

```text
麻婆春雨

麻婆豆腐が作れるなら近い

そのまま使える:
・ひき肉
・豆板醤
・炒める流れ

変わるところ:
・豆腐 → 春雨
・水分量を少し調整

買い足し:
・春雨

[作ってみたい]
[詳しく見る]
```

### 5.5 料理カード詳細画面

目的: 「これなら作れるかも」と思わせる。レシピを全部載せる場所ではなく、**既に作れる料理との差分を理解する画面**。

表示内容: 料理名 / 起点になる料理 / 近さの理由 / そのまま使える知識 / 変えるポイント / 買い足し食材 / ざっくり手順 / 難易度 / 調理時間目安 / 「作ってみたい」ボタン / 保存済み状態。

### 5.6 作ってみたいリスト画面

保存した料理を見返せるようにする。保存は端末内（localStorage）だけでよい。

```text
作ってみたい料理

・麻婆春雨
・鶏チリ
・ツナじゃが
```

空状態:

```text
まだ保存した料理はありません。
気になる料理カードで「作ってみたい」を押してみてください。
```

## 6. 料理カードの仕様

### 6.1 カードの役割

> 既に作れる料理との差分を見せて、「これなら作れそう」と思わせること。

### 6.2 データ型

```ts
type DishCard = {
  id: string
  title: string
  baseDishIds: string[]
  image?: string
  shortCopy: string
  difficulty: 1 | 2 | 3
  timeMinutes: number
  tags: string[]
  reusableSkills: string[]
  changedPoints: string[]
  extraIngredients: string[]
  roughSteps: string[]
  tasteAxis: string[]
  cookingMethod: string[]
  mainIngredients: string[]
}
```

### 6.3 表示する情報

一覧: 料理名 / 「〇〇が作れるなら近い」 / 近い理由1〜2個 / 買い足し食材 / 調理時間 / 難易度 / 作ってみたいボタン。

詳細: 上記に加え、そのまま使えること / 変えるところ / ざっくり手順 / 注意点 / 次に広がる料理候補。

### 6.4 近さの評価軸

1. 味付け
2. 食材
3. 調理方法
4. 工程の軽さ（独立した負荷として扱う）

### 6.5 近さラベル

スコアではなくラベルで表示する: 「かなり近い」「少し変えれば作れる」「ちょっと挑戦」

---

## 7. 作れる料理選択の仕様

- 自由入力させない（表記ゆれ・実装負荷・体験負荷のため）。チップ式選択UI
- 最低1個 / 推奨3個 / 最大制限なし（5個以上で「十分です」表示は任意）
- 選択後「あなたのとなりごはんを探す」ボタンでおすすめ一覧へ

---

## 8. 「作ってみたい」/保存体験

### 8.1 目的

Phase 0で一番重要な行動は、登録ではなく**作ってみたいボタンを押すこと**。料理カードが本当に魅力的だったかの代理指標。

### 8.2 ボタン仕様

未保存: `[作ってみたい]` → 保存後: `[保存済み]`

### 8.3 保存状態

```ts
type SavedDish = {
  dishId: string
  savedAt: string
}
```

`localStorage` に保存。

### 8.4 保存後のフィードバック

「作ってみたいリストに追加しました」＋ `[リストを見る]` を表示。

### 8.5 「作った」記録は必須にしない

詳細画面に小さく `[作った]` を置いてもよいが、優先度は低い。

## 9. 広告・公開方針

- 一般公開できるURLとして出す。「試作版です」と明記してよい
- LPを別に作らず、トップ画面自体を「LP兼Webアプリ入口」にする
- 広告は機能説明よりも悩みに寄せる:

```text
また同じ料理作ってる。
でも新しいレシピを探すのは面倒。

作れる料理から、次の一品を見つける。
となりごはん 試作版
```

- 初期広告予算は数千円単位で十分。少額・短期間・クリエイティブ複数

### 9.5 計測イベント

```text
page_view
start_select_dishes
select_base_dish
show_recommendations
open_dish_card
click_want_to_make
open_saved_list
```

最重要は `click_want_to_make`。個人情報は持たない。

## 10. Go / No-Go基準

### 10.1 判断軸

1. 作れる料理選択まで進むか
2. 料理カードを見るか
3. 詳細を開くか
4. 作ってみたいを押すか
5. 保存一覧を見るか
6. 知人に見せたときに説明なしで理解されるか

### 10.2 Go基準（50〜100セッション程度で）

- 作れる料理選択開始率: 40%以上
- おすすめ表示到達率: 25%以上
- カード詳細閲覧率: 15%以上
- 作ってみたいクリック率: 10%以上
- 1人あたり平均保存数: 0.2以上
- 知人テストで「何をするアプリか」が説明なしで伝わる

### 10.3 No-Go時の分解

| 問題 | 解釈 | 修正 |
| --- | --- | --- |
| トップで離脱 | コピーが弱い | 悩み訴求を強める |
| 料理選択で離脱 | 入力が面倒 | 初期選択を減らす |
| カードで離脱 | 提案が遠い | 近さロジックを修正 |
| 保存されない | 作りたい魅力がない | カード内容を改善 |
| 差別化が伝わらない | UIがレシピアプリ寄り | 「差分」を前面に出す |

No-Goは即終了ではなく、上記の分解で一度修正して再試行する。

## 11. フェーズロードマップ

| Phase | 内容 | 目的 |
| --- | --- | --- |
| Phase 0 | Webアプリ風MVP（広告流入） | 料理の近さ提案が刺さるか確認 |
| Phase 0.5 | 提案品質改善版（データ50件、文言・UI改善） | 料理カードの質を上げる |
| Phase 1 | 継続利用Web版（リスト、履歴、簡易DB、匿名ID） | また使う理由を作る |
| Phase 2 | アプリ前提の体験設計（通知、今日のおすすめ等） | ネイティブ化する価値の確認 |
| Phase 3 | iOS / Androidアプリ化（アカウント、同期、課金） | 継続利用・通知・保存の本格化 |

LINEは Phase 0 の主導線ではない。通知・継続利用の検証が必要になった段階（Phase 2前後）で、補助導線として再検討する。
