# となりごはん 仕様書 v2.8（Draft） — Near-Complete MVP・実利用検証版

最終更新: 2026-06-12
状態: **Draft（オーナーレビュー待ち）**
位置づけ: v2.7（Phase 0 広告検証版）の次サイクルを定義する。意思決定の根拠は `docs/decisions/002-near-complete-mvp-validation.md`。

v2.7 からの主な変更:

- 検証方法を「広告流入の初回反応」から「**実利用とコアループの反復（再訪）**」に変更
- 「作った」記録を必須機能に昇格（v2.7では任意・低優先だった）
- **島マップ**を新規追加。達成感（作った料理が島の領土になる）と探索感（近くの未踏の料理が見える）を担う
- 「今日のとなりごはん」（日替わりおすすめ）を追加し、再訪する理由を作る
- 再訪をPIIなしで計測する仕組みを追加
- 実装はデザインファースト（参照デザイン画像 → Codex実装 → スクリーンショット比較検証）

---

## 1. コンセプト（v2.7から不変）

**となりごはんは、ユーザーがすでに作れる料理を起点に、少し変えれば作れそうな次の料理を提案するレパートリー拡張アプリである。**

コアコピー: 「いつもの料理から、次の一品へ。」

v2.8で加わる視点: レパートリーの広がりを**島マップ**として可視化する。「自分の料理の領土が広がっていく」体験が、達成感と探索感の源になる。

---

## 2. 検証する問い

> ユーザーは、コアループ（選ぶ → 提案を見る → 差分を理解する → 保存する → 作る → 数日内に戻ってくる）を**繰り返す**か？

v2.7の問い（カードを見て作ってみたいと思うか）は前提として引き継ぎ、その先の**定着**を見る。

---

## 3. コアループ（v2.8の中核）

```text
1. 作れる料理を選ぶ（初回のみ。あとから追加・変更可）
2. 「今日のとなりごはん」または近い料理一覧を見る
3. 料理カードで差分を理解する
4. 「作ってみたい」で保存する
5. 実際に作ったら「作った」を記録する
6. 島マップで領土が広がる（達成感・探索感）
7. 数日内にまた開く（再訪）
```

---

## 4. 画面構成

1. トップ画面（LP兼入口 — v2.7を踏襲しつつデザイン刷新）
2. 作れる料理選択画面（v2.7を踏襲。あとから追加できる導線を補強）
3. ホーム画面（**新規**: 今日のとなりごはん + 島マップへの入口 + 保存リストへの入口）
4. おすすめカード一覧画面（v2.7を踏襲）
5. 料理カード詳細画面（v2.7を踏襲 + 「作った」ボタン追加）
6. 作ってみたいリスト画面（v2.7を踏襲 + 作った済み表示）
7. **島マップ画面（新規）**

### 4.1 ホーム画面（新規）

目的: 再訪したユーザーが最初に着地し、「今日はこれ」と1品だけ提示される場所。

- 今日のとなりごはん: 選択済みベース料理から決定的に1品選ぶ（日付シードで同日同結果。API・AI不使用）
- 島マップのサムネイル/入口（領土の現状が一目で見える）
- 作ってみたいリストへの導線（保存件数バッジ）

### 4.2 島マップ画面（新規）

目的: **達成感**（作った料理が自分の領土として塗られる）と**探索感**（隣にまだ見ぬ料理が霞んで見える）。

仕様（最小構成）:

- 料理を「島」上のノードとして配置する。配置は静的データで事前定義（近さの評価軸が近い料理同士を隣接させる）
- ノードの状態は3つ:
  - **作った**（濃い色・領土として確定）
  - **作ってみたい / 作れる**（中間色・足場）
  - **未踏**（霞み表示・タップでカード詳細へ）
- ベース料理として選んだ料理は最初から「作れる」状態で表示
- インタラクションはタップでカード詳細を開くだけ。ドラッグ・ズームは任意（実装コストと相談）
- SVGまたはCanvasで描画。**地図ライブラリ等の新規依存は追加しない**

作らないもの: 自動レイアウト計算 / 3D表現 / アニメーション過多 / 他ユーザーとの比較。

### 4.3 「作った」記録

- カード詳細と保存リストに `[作った]` ボタンを置く
- 記録は localStorage（下記データ型）
- 作った料理は島マップで領土化され、保存リストでは「作った済み」表示に変わる

---

## 5. データ仕様

### 5.1 保存状態（v2.7から拡張）

```ts
type SavedDish = {
  dishId: string
  savedAt: string
  madeAt?: string // 「作った」記録。未記録なら undefined
}
```

### 5.2 島マップ配置データ（新規）

```ts
type IslandNode = {
  dishId: string
  x: number // 0-100 正規化座標
  y: number
  neighbors: string[] // 隣接する dishId
}
```

静的ファイルで管理。生成はビルド時/手動（近さ評価軸ベースで一度設計し、以後は手調整）。

### 5.3 再訪計測（新規・PIIなし）

- localStorage に `tonari.phase0.firstVisitAt`（初回訪問日時）を保存
- 訪問時に前回訪問から24時間以上空いていれば `return_visit` イベントを発火（param: 経過日数のバケット `1-2d` / `3-7d` / `8d+`）
- ユーザーIDは発行しない。横断的なセッション結合はしない

---

## 6. 計測イベント（v2.7の7種 + 3種）

```text
（既存）page_view / start_select_dishes / select_base_dish /
        show_recommendations / open_dish_card / click_want_to_make / open_saved_list
（追加）mark_made        — 作ったボタン（param: dishId）
        open_island_map  — 島マップを開く
        return_visit     — 再訪（param: 経過日数バケット）
```

最重要指標は v2.8 では `mark_made` と `return_visit`。`click_want_to_make` は先行指標に格下げ。

---

## 7. Go / No-Go基準（Draft — 数値はオーナーと要調整）

母数: 配布開始から2〜4週間、ユニーク訪問30〜80人想定（知人・コミュニティ・X/note・少額広告）。

- コアループ完走率（選択→カード→保存）: 30%以上
- `mark_made` 到達: 保存したユーザーの20%以上が1回以上
- `return_visit`: 訪問者の15%以上が3日以内に再訪
- 島マップ: 開いたユーザーの定性反応で「広げたくなる」が観測できる（知人ヒアリング）

No-Go時の分解は v2.7 §10.3 を踏襲し、「作ったが記録されない」（記録動機が弱い）と「再訪がない」（今日のおすすめが弱い／通知がない限界）を追加観点とする。

## 8. デザインファースト・パイプライン（Decision 002）

1. `mocks/` — 文言の正本（画面ごとの日本語コピーをテキストで管理）
2. `mocks/concept/` — ビジュアルの正本（GPT Image等で生成した完成イメージ画像。画面×状態ごと）
3. デザイントークン（色・余白・角丸・影・タイポスケール）を1ファイルに定義し全画面で共有
4. 実装goalごとに: 参照画像を `codex exec -i` で添付 + 正確な日本語コピーをテキストで渡す
5. 検証: 実装画面のスクリーンショットを参照画像と比較（Claude Codeのpreview toolsで実施）

**前提タスク**: `mocks/` と `mocks/concept/` はまだ存在しない。デザイン画像の生成・整理が実装より先。

---

## 9. 作らないもの（v2.8でも変わらず）

ログイン / アカウント / DB / 同期 / LINE / 通知 / 課金 / AI生成レシピ / ユーザー投稿 / 写真アップロード / 栄養計算 / 献立 / ネイティブアプリ。

すべて localStorage 完結。データが端末に閉じることは「試作版です」表記で正直に伝える。

## 10. 実装単位（Codex向け・Draft）

各単位は `docs/goals/` の goal ファイルに対応する。番号は goal 012 から振る。

1. デザイントークン定義 + 全画面への適用基盤（globals.css拡張）
2. ホーム画面（今日のとなりごはん + 各導線）
3. 「作った」記録（SavedDish拡張、ボタン、リスト表示更新）
4. 島マップ配置データ作成（全カード分のIslandNode）
5. 島マップ画面（描画 + 状態表示 + タップ遷移）
6. 再訪計測 + 追加イベント3種
7. トップ画面のデザイン刷新（参照画像ベース）
8. 全画面のデザイン刷新（参照画像ベース、画面ごとに分割可）
9. リリース前チェック（v2.8版チェックリスト）

順序の意図: 機能（2〜6）を先に通し、デザイン刷新（7〜8)は参照画像が揃ってから。デザイン画像の生成が先に終わっていれば並行も可。

## v2.8の最終方針（Draft）

> となりごはん v2.8 は、広告の初回反応ではなく実利用の反復を検証する near-complete MVP として作る。
> ユーザーは作れる料理を起点に提案を受け、作ってみたい料理を保存し、作った料理を記録し、島マップで自分の領土が広がるのを見る。
> 検証するのは「作ってみたい」の先にある「実際に作った」と「数日内にまた開いた」である。
> DB・ログイン・通知は引き続き作らない。localStorageで成立する範囲で、できる限り完成品に見せる。
